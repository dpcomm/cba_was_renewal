# CBA WAS Renewal: 문서 주도 개발(DDD/SDD) 운영 가이드

이 `docs/` 디렉토리는 AI 에이전트(Antigravity 등)와 개발자가 동일한 컨텍스트(Single Source of Truth)를 유지하며 개발하기 위한 **문서 주도 개발 및 스펙 주도 개발**의 핵심 공간입니다.

## 📂 폴더 구조 및 역할

문서는 크게 4가지 목적에 따라 분류됩니다. 작업을 진행할 때 알맞은 위치에 문서를 생성하거나 참조하세요.

### 1. `specs/` (명세서)
- **역할**: 개발(코딩)을 시작하기 전, "무엇을 어떻게 만들 것인가"를 정의하는 스펙 문서입니다.
- **언제 기록하는가**: 새로운 기능(API)을 만들거나 기존 로직을 수정할 때, **코드 작성 전에 가장 먼저** 이곳의 파일을 추가/수정해야 합니다.
- **예시**: `docs/specs/renew_application.spec.md`

### 2. `agents/` (에이전트 헌법 및 프롬프트)
- **역할**: AI 에이전트가 코드를 짤 때 반드시 지켜야 할 원칙(SOP)과 가이드라인입니다.
- **언제 참조하는가**: 에이전트에게 작업을 요청할 때 "어떤 페르소나(규칙)로 코딩할 것인지" 지정하기 위해 사용합니다.
- **예시**: `docs/agents/SPEC_CODER_SOP.md`

### 3. `adr/` (Architecture Decision Records)
- **역할**: 기술 스택, 아키텍처 패턴, 중요한 설계 변경에 대한 의사결정 기록입니다.
- **언제 기록하는가**: 단순히 코드를 수정하는 것을 넘어, 폴더 구조를 통째로 바꾸거나 새로운 라이브러리(패턴)를 도입하는 등 **팀 레벨의 합의(의사결정)가 일어났을 때** 이유와 결과를 기록합니다.
- **예시**: `docs/adr/0001-adopt-ddd-3-layer-architecture.md`

### 4. `history/` (마이그레이션 및 이력)
- **역할**: 과거에 수행되었던 일회성 대규모 작업이나 DB 마이그레이션 이력입니다.
- **언제 기록하는가**: 되돌릴 수 없는 데이터베이스 이관 작업이나 인프라 배포 등 완료된 프로젝트성 결과를 박제할 때 사용합니다.

## 🛠️ 개발 및 AI 운영 워크플로우

개발자(당신)가 AI 에이전트와 코드를 작성할 때는 다음 3단계 워크플로우를 반복합니다.

### 1. 생각은 문서로 (명세서 최신화)
AI에게 코드를 짜달라고 하기 전, 구현하려는 로직의 입출력/예외 케이스/비즈니스 룰을 `docs/specs/` 내의 마크다운 파일에 먼저 업데이트합니다.
- **문서 동기화 시점**: 개발 전 (요구사항 정의), 개발 직후 (설계 변경 시 같은 턴에서 바로 수정)

### 2. 실행은 작업카드로 (에이전트 지시)
AI 에이전트에게 명령할 때, 막연하게 지시하지 않고 **에이전트 헌법(`agents`)과 명세서(`specs`)를 직접 참조(Mention)**시킵니다.
> **프롬프트 예시**:
> _"@[SPEC_CODER_SOP.md] 원칙에 따라서, 방금 업데이트한 @[renew_application.spec.md] 문서의 3번 항목(결제 예외 처리)을 기반으로 `ApplicationService`와 테스트 코드를 작성해줘."_

### 3. 완료는 검증으로 (테스트 및 ADR)
- 코드가 짜여지면 반드시 `npm run test` 를 통해 엣지 케이스 단위 테스트가 통과하는지 확인합니다.
- 만약 AI와 대화하며 기존 아키텍처나 패턴을 크게 바꾸기로 합의했다면, 작업이 끝난 후 `docs/adr/` 에 새로운 기록을 하나 남기고 스프린트를 종료합니다.

> **💡 핵심 요약**: 
> "코드를 짜기 전 스펙(`specs`)을 고치고, AI를 부를 땐 헌법(`agents`)을 들이밀고, 구조를 바꿨다면 결정(`adr`)을 남겨라."
