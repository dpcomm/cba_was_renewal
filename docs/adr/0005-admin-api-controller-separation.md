# ADR 0005: 도메인 모듈 내 관리자 전용 컨트롤러 분리 전략

## Status
**Accepted**

## Date
- **2026-04-30**

## Context (배경)
- 기존의 API 구조는 일반 사용자용 API와 관리자용(Admin) API가 동일한 컨트롤러(예: `UserController`, `ApplicationController`) 안에 혼재되어 있었습니다.
- 이로 인해 개발자가 실수로 `@AdminGuard()`를 누락하여 보안 취약점이 발생할 위험이 컸으며, Swagger 문서 가독성 및 DTO 관리가 어려웠습니다.
- 이를 해결하기 위해 `AdminModule`을 만들고 라우팅을 분리하는 방안을 검토했으나, 동일한 도메인 로직(Service)을 공유해야 하는 특성상 모듈 간 순환 참조(Circular Dependency) 및 과도한 보일러플레이트 코드 문제가 우려되었습니다.

## Decision (결정)
- **컨트롤러 레벨 분리**: 별도의 Admin 모듈을 구성하지 않고, 기존 도메인 모듈 안에서 관리자용 컨트롤러를 분리하여 생성합니다. (예: `UserAdminController`)
- **라우팅 Prefix**: 관리자 전용 컨트롤러에는 `@Controller('admin/users')` 와 같이 `admin/` 접두사를 일괄 적용합니다.
- **클래스 레벨 가드**: 분리된 관리자 컨트롤러의 최상단에 `@AdminGuard()`를 선언하여, 내부의 모든 메서드가 예외 없이 보안 검사를 받도록 강제합니다.

## Consequences (결과)
### 장점 (Pros)
- 인가(Authorization) 로직이 컨트롤러 클래스 단위로 격리되어 보안 사고 위험이 원천 차단됩니다.
- 도메인 단위의 모듈 응집도(User 도메인의 코드는 UserModule에 모여있음)를 그대로 유지하면서 라우팅만 우아하게 분리할 수 있습니다.
- NestJS의 순환 참조 문제를 회피할 수 있어 아키텍처가 단순해집니다.

### 단점 (Cons)
- 일반 사용자 컨트롤러와 관리자 컨트롤러 두 개를 각각 생성하고 모듈에 등록해야 하므로 파일 개수가 소폭 증가합니다.
- 관리자 전용의 글로벌 미들웨어를 걸거나 일괄 설정을 하려면 라우트 접두사(`admin/*`) 기반의 설정이 필요합니다.
