# ADR 0007: Rich Domain Entity 및 UseCase/Query 패턴 도입

## Status

**Accepted**

## Date
- **2026-04** (유저 모듈 리팩토링 및 백오피스 고도화 단계)

## Context (배경)

- 초기에 채택한 3계층 아키텍처(ADR 0001)에 따라 `Service` 클래스에 비즈니스 로직을 모아두었으나, 프로젝트가 성장함에 따라 `UserService`와 같은 단일 파일이 수백 줄이 넘어가는 **God Object(Fat Service)**가 되는 문제가 발생했습니다.
- 여러 기능(조회, 수정, 탈퇴, 이메일 변경 등)이 하나의 파일에 묶여 있어 가독성이 떨어지고, 의존성 주입이 비대해지며, 특정 로직을 수정할 때 사이드 이펙트를 예측하기 어려웠습니다.
- 엔티티(`Entity`)는 데이터베이스 테이블을 매핑하는 역할만 하고 비즈니스 로직이 없는 **Anemic Domain Model(빈약한 도메인 모델)** 상태였습니다.

## Decision (결정)

유지보수성과 가독성을 높이기 위해, Application 및 Domain 계층의 설계 방식을 다음과 같이 변경합니다.

1. **Rich Domain Entity (풍부한 도메인 모델)**
   - 자신의 상태를 변경하는 비즈니스 로직과 검증 규칙은 모두 `Entity` 내부의 도메인 메서드(예: `changeEmail()`, `softDelete()`)로 캡슐화합니다.
   - 외부 인프라(DB, 외부 API) 의존성 없이, 순수하게 파라미터와 자신의 상태만으로 비즈니스 규칙을 처리합니다.

2. **Application 계층의 분리 (UseCase & Query)**
   - 거대한 단일 `Service` 클래스를 폐기하고, 오직 하나의 책임만 가지는 여러 클래스로 쪼갭니다.
   - **Query (조회 전용)**: DB에서 데이터를 단순히 읽어오는 역할 (예: `GetUserQuery`, `SearchUsersQuery`).
   - **UseCase (명령/실행 전용)**: DB에서 엔티티를 불러오고, 엔티티의 도메인 메서드를 호출하여 상태를 변경한 뒤 영속화하는 로직의 흐름(오케스트레이션)을 담당합니다 (예: `DeleteAccountUseCase`).

## Consequences (결과)

### 장점 (Pros)
- 클래스 하나당 코드가 짧아지고 책임이 명확해져 유지보수성과 가독성이 크게 향상됩니다. (Single Responsibility Principle)
- 특정 컨트롤러가 필요한 로직만 정확히 주입(`@Inject`)받을 수 있어 결합도가 낮아집니다.
- AI 에이전트와 협업 시, 파일 크기가 작고 목적이 명확하여 더욱 빠르고 정확한 코드 생성이 가능합니다.
- 복잡한 도메인 논리가 엔티티에 캡슐화되어 비즈니스 규칙을 파악하기 쉽습니다.

### 단점 (Cons)
- 하나의 기능을 파악하기 위해 여러 개의 쪼개진 파일(`Controller -> UseCase -> Entity`)을 넘나들며 추적해야 합니다.
- 파일의 개수가 이전보다 훨씬 많아져 초기 보일러플레이트 작성 비용이 증가합니다.
