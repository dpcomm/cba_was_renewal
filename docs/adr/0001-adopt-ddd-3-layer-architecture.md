# ADR 0001: DDD 기반 3계층 아키텍처 및 단방향 의존성 규칙 도입

## Status

**Accepted**

## Date
- **2025-11** (프로젝트 초기 설계 단계)

## Context (배경)

- CBA WAS 프로젝트를 진행하며 기존 레거시 코드의 유지보수 어려움(스파게티 코드, 비즈니스 로직과 웹 계층의 결합)을 해결해야 했습니다.
- 어떤 계층에서 무엇을 처리해야 하는지 명확한 가이드라인이 없으면 팀원(또는 AI)마다 코딩 스타일이 파편화될 우려가 있었습니다.

## Decision (결정)

- 도메인 중심 설계(DDD) 사상을 차용하여 모든 모듈을 **Presentation(표현) -> Application(응용) -> Domain(도메인)**의 3계층으로 분리합니다.
- **엄격한 단방향 의존성 규칙**을 강제합니다:
  - 하위 계층은 상위 계층을 절대 알지 못해야 합니다. (역참조 금지)
  - `Service`(Application)는 HTTP 객체나 `Controller`(Presentation)의 DTO를 직접 알면 안 됩니다.
  - `Entity`(Domain)는 다른 계층의 로직이나 포맷에 의존하지 않고 순수해야 합니다.

## Consequences (결과)

### 장점 (Pros)

- 비즈니스 로직(Application/Domain)이 특정 웹 프레임워크나 HTTP 포맷에 종속되지 않아, 추후 변경이나 단위 테스트가 매우 쉬워집니다.
- 코드의 역할이 명확해져 유지보수성과 가독성이 크게 향상됩니다.

### 단점 (Cons)

- 계층을 나누다 보니 DTO, Mapper, Entity 간의 데이터 변환 보일러플레이트 코드가 증가합니다.
