# ADR 0002: Express에서 NestJS로의 점진적 마이그레이션 및 트래픽 분리 (Strangler Fig Pattern)

## Status

**Accepted**

## Date
- **2025-11 ~ 2026-01** (점진적 마이그레이션 진행 기간)

## Context (배경)

- 기존 CBA Connect 백엔드는 Express 서버로 구축되어 있었으나, 비즈니스 로직과 Prisma 스키마가 강결합되어 있어 기능 추가 및 확장이 어려운 상태였습니다.
- 이를 해결하기 위해 NestJS + TypeORM 기반의 새로운 서버(WAS Renewal)를 구축하기로 결정했습니다.
- 그러나 방대한 기존 시스템을 한 번에 전환(Big Bang)하는 것은 서비스 중단 및 버그 발생 리스크가 매우 컸습니다.

## Decision (결정)

- **점진적 마이그레이션(Strangler Fig Pattern) 도입**: 운영 중인 레거시 서버(Express)를 중단하지 않으면서, 새로운 기능이나 리팩토링이 완료된 도메인부터 순차적으로 NestJS로 이관합니다.
- **Nginx Ingress를 통한 트래픽 분리**: 오라클 클라우드의 VC 기반 온프레미스 쿠버네티스(k8s) 환경에서 Nginx Ingress 컨트롤러를 활용합니다. API 라우트 프리픽스(경로)를 기반으로, 아직 이관되지 않은 구형 API는 Express 서버 파드로, 새로 구현된 API(`/api/v2/...` 등)는 NestJS 서버 파드로 트래픽을 분기합니다.

## Consequences (결과)

### 장점 (Pros)

- 마이그레이션 과정에서도 서비스 다운타임 없이 안정적으로 프로덕션을 운영할 수 있습니다.
- 새로운 인프라나 코드가 잘못되었을 때 전체 장애로 번지지 않고, 문제의 라우트만 Express로 원복할 수 있는 안전장치가 마련됩니다.

### 단점 (Cons)

- 과도기 동안 두 개의 백엔드 프로젝트와 DB 컨넥션을 동시에 운영하고 모니터링해야 하는 관리 비용이 발생합니다.
- 공통으로 사용하는 세션이나 캐시(Redis 등), 인증 정책을 두 서버 간에 동일하게 동기화해야 합니다.
