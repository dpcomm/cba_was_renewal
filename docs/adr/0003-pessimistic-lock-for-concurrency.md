# ADR 0003: 선착순 기능 동시성 이슈 해결을 위한 비관적 락(Pessimistic Lock) 도입

## Status

**Accepted**

## Date
- **2026-01** (겨울수련회 전 동시성 방어 로직 적용)

## Context (배경)

- 대형 수련회 환경(400명 이상)에서는 "선택식 강의 신청"이나 "웰컴 이벤트 당첨(룰렛)"과 같은 기능에 짧은 시간 동안 트래픽이 집중됩니다.
- 한정된 수량(정원, 당첨자 수 제한)을 두고 여러 사용자가 동시에 API를 호출할 때, 일반적인 트랜잭션 처리만으로는 갱신 손실(Lost Update)로 인해 정원을 초과하여 신청되는 동시성 문제(Race Condition)가 발생할 위험이 있습니다.

## Decision (결정)

- **비관적 락(Pessimistic Lock) 적용**: 동시 요청 시 데이터 무결성을 확실하게 보장하기 위해, 트랜잭션 내에서 데이터를 읽어올 때 DB 레벨의 배타적 락(`FOR UPDATE`, TypeORM의 `lock: { mode: 'pessimistic_write' }`)을 적용합니다.
- 선택식 강의 신청, QR 체크인 기반 이벤트 참여 등 **정원 제한이 있는 핵심 로직**에만 선별적으로 적용합니다.

## Consequences (결과)

### 장점 (Pros)

- 동시에 들어오는 초과 신청이나 이벤트 중복 당첨을 완벽하게 차단하여 데이터 정합성을 100% 보장합니다.
- 낙관적 락(Optimistic Lock) 구현 시 충돌 발생 후 재시도 처리 로직을 구현해야 하는 복잡함을 덜고 직관적으로 코드를 작성할 수 있습니다.

### 단점 (Cons)

- 한 트랜잭션이 끝날 때까지 다른 트랜잭션이 대기(Blocking)해야 하므로 처리량이 극단적으로 몰릴 경우 성능 저하(병목 현상)나 데드락 위험이 있습니다. (이를 방지하기 위해 락 유지 시간을 최소화하도록 로직을 짧게 설계해야 합니다)
