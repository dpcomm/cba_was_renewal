# ADR 0004: 데이터 보존형 SQL 마이그레이션 전략

## Status
**Accepted**

## Date
- **2026-04-01** (수련회 데이터베이스 리뉴얼 이관)

## Context (배경)
- 기존 수련회 신청 시스템의 파편화된 데이터(`surveyData` JSON 필드 등)를 새로운 스키마 구조로 이관해야 했습니다.
- 초기에는 별도의 TS 스크립트(`migrate-survey-data.ts`)를 만들어 어플리케이션 단에서 데이터를 옮기려 했으나, 운영 환경(Production)에서 중간에 에러가 났을 때 재시도가 불가능하거나 데이터가 유실될 위험성이 매우 높았습니다.

## Decision (결정)
- 어플리케이션 스크립트 방식 대신, **TypeORM 기반의 수동 SQL 마이그레이션 방식**을 채택합니다.
- `DROP` & `ADD` 방식의 자동 생성 마이그레이션을 피하고, `ALTER TABLE ... CHANGE COLUMN` 명령을 직접 사용하여 **기존 신청 데이터를 100% 보존하며 컬럼명과 속성만 변경**하는 "데이터 보전형 이관"을 원칙으로 삼습니다.
- 마이그레이션 파일 내부에는 `NOT NULL` 추가 시 발생하는 에러를 방지하기 위해, 사전에 NULL 데이터를 기본값으로 채워넣는 방어 로직과 멱등성(Idempotent, 재시도 가능성) 보장 로직을 반드시 포함합니다.

## Consequences (결과)
### 장점 (Pros)
- 마이그레이션 실패 시 언제든 롤백(`revert`)하고 원인을 파악한 뒤 재시도할 수 있는 안정성이 확보되었습니다.
- 단일 트랜잭션 내에서 스키마 변경과 데이터 정합성 보정이 동시에 일어나므로, 운영 DB의 다운타임을 최소화할 수 있습니다.

### 단점 (Cons)
- TypeORM이 자동으로 생성해주는 마이그레이션 코드를 그대로 쓰지 못하고, 개발자(또는 AI)가 SQL과 실행 순서를 꼼꼼히 검토하고 수동으로 최적화(수정)해야 하는 공수가 듭니다.
